  理清现状、目标与阻碍，是避免开发方向偏离的唯一方法。                                                                                                          我将通过阅读现有文件，结合我们刚才的深度讨论，为您整理出一份**《Windows

  🗺️ Windows Computer Use MCP 项目全景快照 (Project Snapshot)                …

  1. 项目愿景 (Project Vision)

  构建一个高精度的 MCP 服务器，使 AI 智能体能够通过标准协议，实现对 Windows
  桌面环境的像素级操控。其核心价值在于：让智能体看到的屏幕（Logical
  View）与它实际能操作的屏幕（Physical View）在坐标空间上实现完全一致。

  2. 核心技术栈 (Core Tech Stack)

  - 控制协议: Model Context Protocol (MCP)
  - 编排层 (Orchestration): TypeScript (Node.js) ——
  负责业务逻辑、坐标转换算法、工具分发。
  - 执行层 (Execution): Rust (NAPI-RS) —— 高性能、低延迟地调用 Windows 底层
  API。
  - 驱动层 (Driver): enigo (键鼠)、dxgi (截图)、windows-rs (窗口管理)。

  3. 架构模型 (Architecture Model)

  graph TD
      A[AI Agent / Claude Code] -- JSON-RPC --> B[MCP Layer: index.ts]
      B -- Method Call --> C[Executor Layer: executor.ts]
      C -- Coordinate Conversion --> D[Native Bridge: napi-rs]
      D -- Low-level API --> E[Windows OS: enigo/dxgi/user32]
      E -- Hardware Action --> F[Mouse/Keyboard/Display]

  4. 当前核心矛盾 (The Core Conflict)

  “坐标系失真” (Coordinate Inconsistency)
  - 现象：智能体发出的逻辑坐标在执行时，鼠标移动位置与预期不符（如：移动到左上
  角），导致无法点击目标窗口，进而导致 type 指令失效。
  - 根源：在多显示器 + 异构 DPI
  缩放的环境下，当前的坐标转换模型过于简化（仅使用单一缩放因子），未能处理
  Windows 虚拟桌面坐标系中的偏移量 (Offset) 和 多屏 DPI 差异。

  5. 关键技术挑战 (Technical Challenges)

  - [高优先级]
  全局坐标对齐：如何获取当前鼠标/窗口所在的显示器及其在全局虚拟坐标系中的绝对
  (left, top) 偏移量。
  - [高优先级] DPI 感知同步：确保 Rust 层获取的 DPI 信息与 Windows 系统的
  SetProcessDpiAwareness 状态完全匹配，避免 enigo 的 Abs 模式产生二次缩放。
  - [中优先级] 状态同步：确保 open_app 与 focus_window 之间的时序逻辑（Race
  Condition）足够稳健，保证输入前焦点已锁定。

  6. 路线图 (Roadmap)

  第一阶段：建立“绝对坐标引擎” (The Precision Engine)

  - 目标：解决“点不准”的问题。
  - 动作：
    - Rust 层：扩展 WindowManager API，返回目标显示器的 MonitorRect (全局偏移)
  和 DPI Scale。
    - TS 层：重构 logicalToPhysical 算法，公式改为：Physical = Monitor_Origin +
   (Logical * Scale)。

  第二阶段：强化“焦点控制” (The Focus Guarantee)

  - 目标：解决“打不进”的问题。
  - 动作：
    - 实现基于 SetForegroundWindow 的窗口激活逻辑。
    - 将 open_app 升级为“启动 $\rightarrow$ 轮询 $\rightarrow$
  强制激活”的原子操作。

  第三阶段：工程化与稳定性 (Hardening)

  - 目标：解决“不可靠”的问题。
  - 动作：完善错误处理、增加集成测试套件、优化 DXGI 截图性能。
