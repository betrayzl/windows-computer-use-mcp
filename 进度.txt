  1. 系统架构全景视图 (System Architecture Overview)

  目前项目采用的是**“分层驱动 + 实例映射”**的架构模式。

  逻辑拓扑结构

  1. Protocol Layer (MCP Server): 基于 stdio 的通信协议层，负责解析 JSON-RPC
  消息。
  2. Orchestration Layer (TS Executor):
  核心业务逻辑层。它不直接操作底层，而是维护了三个关键的 Rust
  实例：ScreenCapture、InputController、WindowManager。
  3. Native Bridge Layer (NAPI-RS): 跨语言转换层。通过 napi-rs 将 Rust
  的内存对象映射为 JS 对象。
  4. Hardware/OS Interaction Layer (Rust Native): 最终执行层。直接与 Windows API
   (DXGI, User32, Gdi, Enigo) 进行交互。

  数据流向图 (Data Flow Path)

  - 控制流 (Control Flow): Client -> MCP -> Executor -> NAPI -> Rust Method ->
  Windows API。
  - 数据流 (Data Flow - 以截图为例): Windows Surface -> DXGI Texture -> Rust
  Buffer (RGBA) -> Base64 String -> Node.js String -> MCP Response。

  2. 核心问题诊断 (Critical Issues Matrix)

  通过对代码的静态分析，我识别出以下三个维度的风险点：

  A. 性能与阻塞风险 (Performance & Blocking) —— [致命]

  - 现象：test_driver.js 在 screenshot 阶段出现 Timeout。
  - 根源分析：
    - 线程模型错配：ScreenCapture::do_capture 在 Rust
  层是一个巨大的同步块。虽然它在 TS 层被标记为 async，但由于它在 NAPI 默认的
  Main Thread 上运行，它会长时间霸占 Node.js 的事件循环。
    - 计算密集型任务过重：do_capture 内部包含了：AcquireNextFrame (等待 I/O)
  $\rightarrow$ CopyResource (GPU 内存拷贝) $\rightarrow$ Map (内存映射)
  $\rightarrow$ 逐字节像素循环转换 (CPU 密集) $\rightarrow$ JPEG 编码 (CPU 密集)
   $\rightarrow$ Base64 编码 (CPU 密集)。
    - 后果：在整个转换过程中，Node.js 的 stdio 无法处理任何心跳或后续指令，导致
  MCP 客户端认为服务器已死。

  B. 精度与一致性风险 (Accuracy & Consistency) —— [高]

  - 现象：get_display_size 返回硬编码值，且 scale_factor 始终为 1.0。
  - 根源分析：native/src/window.rs 处于占位状态。
  - 后果：这会导致“坐标漂移”。如果用户屏幕是 2K 且开启了 150% 缩放，而程序按
  1080p/100% 计算，所有的鼠标点击（move_mouse, click）都会点在错误的位置。

  C. 资源管理风险 (Resource Management) —— [中]

  - 现象：FrameGuard 使用了 Drop trait 来释放 DXGI 帧。
  - 潜在风险：如果在 do_capture 中发生 panic（例如内存溢出或非法指针访问），Drop
   可能不会被触发，导致 IDXGIOutputDuplication 句柄未释放，后续调用将持续失败。

  3. 演进设计方案 (Evolutionary Design)

  为了将项目从“功能原型”提升到“生产级工具”，我设计了以下三个阶段的演进方案。

  第一阶段：重构执行引擎 (Refactoring the Engine)

  目标：解决 Timeout 问题，实现真正的异步。

  1. 引入 NAPI 异步任务模型：
    - 将 do_capture 的核心逻辑封装进
  napi::Task。这样，耗时的计算（像素转换、JPEG 编码）将在 Node.js 的 Worker Pool
   中运行，不会阻塞主线程。
  2. 优化像素处理算法 (SIMD/Bulk Copy)：
    - 废弃当前的 for 循环逐字节 push 方式。
    - 引入 std::ptr::copy_nonoverlapping 进行内存块拷贝。
    - 优化颜色转换：利用 image crate 的 convert
  功能，或直接在内存中进行位移操作，减少 CPU 分支预测失败。
  3. 改进 DXGI 状态机：
    - 实现更健壮的 AcquireNextFrame 重试机制，增加 max_retries 和指数退避逻辑。

  第二阶段：实现环境感知 (Environment Awareness)

  目标：解决坐标漂移问题。

  1. 接入真实 DPI 系统：
    - 在 native/src/window.rs 中调用 GetDpiForMonitor。
    - 返回 DisplayGeometry { width, height, scale_factor }。
  2. 完善窗口管理 (Window Management)：
    - 实现基于 EnumWindows 和 GetWindowThreadProcessId 的进程匹配逻辑。
    - 通过 ShowWindow(hwnd, SW_HIDE/SW_SHOW) 实现可靠的窗口隐藏/恢复。

  第三阶段：工程化加固 (Hardening & Packaging)

  目标：提升稳定性与易用性。

  1. 完善错误处理体系：
    - 建立统一的 NativeError 枚举，确保所有 Windows API 的 HRESULT
  都能转化为具有语义的 JavaScript Error。
  2. 构建自动化流水线：
    - 完善 build.js，确保从 Rust 编译到 ncc 打包的一键式体验。

  4. 总结与下一步建议

  当前项目的核心矛盾在于：用“同步的思维”在写“异步的接口”，导致了严重的线程阻塞。

  我的建议动作：
  不要尝试通过增加超时时间来“掩盖”问题，而应该直接进入 Rust 层的异步重构。

  如果您准备好了，我建议我们的第一个动作是：
  ▎ 重构 native/src/capture.rs，将 do_capture 迁移到 napi::Task
  模式，并优化像素拷贝逻辑。

                                                                                  项目进度总结：Windows Computer Use MCP                                        
  1. 项目背景与目标                                                             
  本项目是一个基于 Model Context Protocol (MCP) 的 Windows 自动化服务器。
  - 核心能力：通过 Rust (NAPI-RS) 调用 Windows 底层 API，实现屏幕截图
  (DXGI)、键鼠模拟 (Enigo)、窗口管理及剪贴板操作。
  - 技术栈：Rust (Native 层) + TypeScript (MCP/Executor 层) + Node.js (运行时)。

  2. 已解决的核心问题 (Milestones)

  阶段一：基础功能与架构搭建

  - 实现了分层架构：MCP Layer $\rightarrow$ Executor Layer $\rightarrow$ Native
  Layer $\rightarrow$ Loader Layer。
  - 完成了完整的构建流水线：node build.js 一键完成 Rust 编译、TS 编译与 Bundle
  打包。

  阶段二：解决致命的“状态破坏”与“线程挂起”问题

  在集成 capture_excluding
  功能时，我们发现了严重的稳定性缺陷，并通过以下方案彻底解决：
  - 问题发现：在 capture_excluding 流程中，由于在 EnumWindows
  同步回调中执行了高成本的 OpenProcess 和进程名匹配，导致了严重的线程阻塞。同时
  ，若截图失败，被隐藏的窗口无法恢复，造成了系统状态破坏。
  - 架构重构 (RAII 模式)：
    - 分离查询逻辑：将 EnumWindows 回调改为轻量级的“仅收集句柄与
  PID”，将耗时的进程匹配逻辑移至回调之外。
    - 引入 WindowRestoreGuard：利用 Rust 的 Drop trait 实现 RAII
  模式，确保无论截图过程是成功还是出错，被隐藏的窗口都能自动恢复。
  - 类型安全升级：将所有涉及窗口句柄 (HWND) 的传输类型从 i32 升级为
  i64，彻底解决了 64 位系统下的句柄截断隐患。
